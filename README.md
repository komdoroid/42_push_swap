# push_swap

42Tokyo カリキュラムの push_swap 課題（バージョン1.1）。
2つのスタックと限られた操作だけを使って整数列を昇順に並べ替える「操作手順」を生成するプログラム。

---

## 1. Description（概要）

push_swap は、整数列をソートするための **操作手順** を標準出力に出力するプログラムです。
実際にソートされた結果ではなく、「この手順を順に実行すればsoートできる」という操作列（`ra`, `pb` など）を出力します。

本プログラムは4つのソート戦略を1つのバイナリに実装しています。

| 戦略 | 複雑度クラス | 使用アルゴリズム |
|------|------------|----------------|
| Simple | O(n²) | 挿入ソート（b退避型） |
| Medium | O(n√n) | チャンクソート（√n分割） |
| Complex | O(n log n) | 基数ソート（LSD） |
| Adaptive | 入力次第 | disorder に応じて上3つを切替（デフォルト） |

### データ構造

スタックは **双方向循環連結リスト** で実装しています。
全ての操作（sa/sb/ss/pa/pb/ra/rb/rr/rra/rrb/rrr）を O(1) で実行でき、
操作の意図とコードが1対1で対応するため、保守性と説明のしやすさを重視してこの構造を選びました。

リングバッファも全操作 O(1) で実現できますが、要素移動のたびにインデックス計算・
マイナス補正・サイズ更新が必要でコードが分散するため、双方向循環連結リストを採用しました。

## 2. Instructions（使い方）

### ビルド

```
make
```

### 基本的な使い方

```
./push_swap 3 1 2 5 4
```

整数を引数として渡すと、ソート手順を標準出力に出力します。

文字列形式（引数1個に複数の数値）にも対応しています。

```
./push_swap "3 1 2 5 4"
```

### 戦略フラグ

ソート戦略を明示的に指定できます。フラグは省略可能で、省略時は Adaptive が選ばれます。

```
./push_swap --simple 3 1 2     # O(n²) を強制
./push_swap --medium 3 1 2     # O(n√n) を強制
./push_swap --complex 3 1 2    # O(n log n) を強制
./push_swap --adaptive 3 1 2   # disorder に応じて自動選択（デフォルト）
```

戦略フラグは互いに排他です（2つ以上指定するとエラー）。

### ベンチマークモード

`--bench` を付けると、ソート完了後に実行統計を **標準エラー出力** に出します。
戦略フラグと併用できます。

```
./push_swap --bench 3 1 2
./push_swap --simple --bench 3 1 2
```

標準出力（操作列）と標準エラー出力（統計）が分離されているため、
checker と組み合わせながら統計を確認できます。

```
./push_swap --bench 3 1 2 2>bench.txt | ./checker 3 1 2
```

### エラーと特殊な入力

以下はエラーとして `Error\n` を標準エラー出力に出します。

- 整数以外の文字を含む
- int の範囲外（INT_MAX 超過、INT_MIN 未満）
- 重複する値
- 不正なフラグ、戦略フラグの複数指定

以下は何も出力せず正常終了します。

- 引数なし
- 既にソート済みの入力

## 3. アルゴリズムと複雑度の論証

> ※ ここでの複雑度は、生成される **push_swap 操作の数** で測っています。
> 配列ベースの古典的アルゴリズムの理論的複雑度ではありません。

### Simple: 挿入ソート（O(n²)）

a から最小値を順に b へ退避し、b を降順に保ちます。最後に b から a へ戻すと昇順になります。
要素を1つ正しい位置に置くのに最悪 O(n) の操作、それを n 要素分行うため O(n²)。

### Medium: チャンクソート（O(n√n)）

値の範囲を √n 個のチャンクに分割し、a から各チャンクを順に b へ振り分けます。
b はチャンクの塊単位で降順になり、戻すときの回転コストが抑えられます。

総コスト f(k) = kn + n²/k（k はチャンク数）を最小化すると k = √n となり、
このとき総コストは 2n√n = O(n√n)。

> ※ 実装の操作数が本当にこのオーダーに収まっているかは、`--bench` を用いて
> 複数サイズで測定し確認しています（n を4倍にしたとき操作数が約8倍になることを確認）。
> 〔測定データをここに記載する〕

### Complex: 基数ソート（O(n log n)）

全要素にインデックス（0〜n-1）を振り、下位ビットから順に処理します。
各ビットについて「ビットが0なら b へ退避、1なら a に残す」を繰り返し、
b を a に戻すことを ⌈log₂(n)⌉ 回行います。

各ラウンドが O(n)、ラウンド数が O(log n) のため、全体で O(n log n)。
インデックス化により、扱う値の範囲を n に圧縮し、必要ビット数を最小化しています。

### Adaptive: 戦略選択

ソート前に disorder（乱れ度）を1度だけ計算し、しきい値で戦略を切り替えます。

```
disorder = (逆転対の数) / (全ペア数 n(n-1)/2)
```

| disorder | 選択戦略 |
|----------|---------|
| < 0.2 | Simple |
| 0.2 〜 0.5 | Medium |
| ≥ 0.5 | Complex |

> ※ しきい値の根拠と各戦略の使い分けについては〔ここに記載〕。

## 5. AI の使用について

〔課題要件として、AI をどう使ったかを正直に記載する〕

本プロジェクトの **設計検討と概念理解** において AI（Claude）を壁打ち相手として利用しました。
具体的な利用範囲は以下の通りです。

- データ構造（双方向循環連結リスト vs リングバッファ vs 配列）の比較検討
- 各ソートアルゴリズムの選定理由とトレードオフの議論
- パース処理・エラー処理の設計方針の検討
- 整数オーバーフロー（2の補数の非対称性、INT_MIN の扱い）の理解
- メモリ管理（malloc 要否、初期化、循環リストの解放）の理解

**コードの記述はすべて自分（およびペア）が行いました。**
AI の提案は鵜呑みにせず、各判断について自分で検証し、
誤りや不適切な提案（不要な free の混入、エラー処理の見落としなど）は自分で指摘・修正しました。
