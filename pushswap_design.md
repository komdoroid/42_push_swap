# push_swap 設計ドキュメント

このドキュメントは、push_swap課題に取り組むにあたっての**設計判断とその根拠**をまとめたものです。
実装を始める前に合意しておくべき方針を整理しています。

ドキュメントの構成:
- 1〜2章は前提知識と全体像
- 3章が実装の本体。**書く順番に並べてある**ので上から順に取り組める
- 4〜5章は採用判断の詳細な根拠（レビュー対策）
- 6〜8章は周辺方針と進め方

各セクションでは「何をやる必要があるか」「採用する方針」の流れで記述し、詳細な根拠は4〜5章にまとめています。

---

## 1. 課題の概要と合格基準

今回のpush_swapはカリキュラム改定前のものより要求が多い課題です。

- **4つの戦略を1つのバイナリ(実行ファイル)に実装する**
  - Simple: O(n²)
  - Medium: O(n√n)
  - Complex: O(n log n)
  - Adaptive: disorderに応じて上3つを使い分け
- 起動時のフラグ（`--simple` / `--medium` / `--complex` / `--adaptive`）でアルゴリズムを切り替え
- フラグなしの場合は `--adaptive` がデフォルト
- `--bench` オプションで実行統計をstderrに出力（戦略フラグと併用可能）
- 2人組課題

### 合格基準（操作数）

| 入力サイズ | 合格 | 良好 | 優秀 |
|-----------|------|------|------|
| 100要素 | < 2000 | < 1500 | < 700 |
| 500要素 | < 12000 | < 8000 | < 5500 |

## 2. 前提知識

実装に入る前に、後の章で頻出する概念を押さえておく。

### Big-O記法の直感

「操作回数がnに対してどう増えるか」を表す階級表記。具体的な手数ではなく**増え方の傾向**を見る。

　-ビッグオー記法について　（https://zenn.dev/shoan/articles/d13f3dde7c8400

| 表記 | 増え方 | n=100 | n=500 | 例 |
|------|--------|-------|-------|------|
| O(1) | 一定 | 1 | 1 | 配列の `arr[5]` |
| O(log n) | ゆるやか | 約7 | 約9 | 二分探索 |
| O(n) | 比例 | 100 | 500 | 配列を全走査 |
| O(n log n) | やや急 | 約700 | 約4500 | マージソート, 基数ソート |
| O(n√n) | 急 | 1000 | 約11000 | チャンクソート |
| O(n²) | 爆発的 | 10000 | 250000 | バブルソート |

push_swapの合格基準と照らすと、500要素では**O(n log n)階級でなければ届かない**ことが分かる。

### push_swap言語の11個の操作

| 操作 | 動作 |
|------|------|
| `sa` | aの先頭2つをswap |
| `sb` | bの先頭2つをswap |
| `ss` | saとsbを同時実行 |
| `pa` | bの先頭をaの先頭に移動 |
| `pb` | aの先頭をbの先頭に移動 |
| `ra` | aを1つ回転（先頭→末尾） |
| `rb` | bを1つ回転 |
| `rr` | raとrbを同時実行 |
| `rra` | aを逆回転（末尾→先頭） |
| `rrb` | bを逆回転 |
| `rrr` | rraとrrbを同時実行 |

これら全てを**O(1)で実現できるデータ構造**を選ぶ必要がある。詳細は4章で比較する。

## 3. 実装ステップ

実装は以下の順番で進める。前のステップで作ったものを次のステップで使うため、原則この順序を崩さない。

### 3-1. パース処理

#### やること

コマンドライン引数を受け取り、整数のリストに変換する。入力エラーは全て検出して `Error\n` をstderrに出力して終了する。

#### 受け付ける入力

- `int` の範囲内の整数（負数含む）
- 引数1個に複数の数字 (`"1 2 3"`) も1引数として処理（空白でsplit）

#### エラーとして弾く入力

- 数字以外の文字を含む
- `INT_MAX` を超える / `INT_MIN` を下回る
- 重複する値
- 空文字列 `""`
- 数値として解釈できない文字列 (`"one"`, `"--5"` など)

#### エラーではない入力（何も出力せず正常終了）

- 引数ゼロ個
- 引数1個で単一の数字 (`./push_swap 42`)
- 既にソート済みの入力（例: `./push_swap 1 2 3 4 5`）

#### 実装時の注意

- パース時は `long` で受けて、`int` の範囲チェックを行う
- 重複検出は二重ループ（O(n²)）で十分（パース時のみのため）
- エラー時の出力先は **stderr** (fd=2)。stdoutに出すとcheckerと組み合わせたときに誤動作する
- フラグ引数（`--simple` など）は整数パースの対象外として最初に分離する

---

### 3-2. データ構造（スタックとノード）

#### やること

push_swapの11個の操作を全て効率的に実行できるデータ構造を実装する。回転系（ra/rb/rra/rrb）はソート中に大量に発生するため、O(1)で実現できることが必須。

#### 採用方針: 双方向循環連結リスト

詳細な比較と採用根拠は **4章** を参照。

#### 必要な構造体

- **ノード**: `value` (int) と `prev` / `next` ポインタを持つ
- **スタック**: `top` ポインタ（先頭ノードを指す）と `size` (int) を持つ

#### 循環の性質

- 末尾ノードの `next` は先頭ノード（`top`）を指す
- 先頭ノードの `prev` は末尾ノードを指す
- これにより `top->prev` で末尾に O(1) で到達できる

#### 必要な補助関数

- スタック初期化（空のスタックを作る）
- ノードの作成（値を1つ受け取って新規ノードを返す）
- スタック全体の解放（全ノードを辿ってfree、最後にスタック構造体もfree）
- サイズ取得、min/max検索（ソート中に使う）

---

### 3-3. 11個の操作関数

#### やること

push_swap言語の11個の操作を、双方向循環連結リスト上で実装する。**全てO(1)** で実現する。

#### 各操作の概念的な動作

| 操作 | 実装の方針 |
|------|----------|
| `sa` / `sb` | 先頭2ノードの値を入れ替える、または前後関係を繋ぎ替える |
| `ss` | sa と sb を順に呼ぶ |
| `pa` / `pb` | 片方のスタックの先頭ノードをdetachし、もう片方の先頭にattach |
| `ra` / `rb` | top を `top->next` に進める（循環しているのでこれだけで完了） |
| `rr` | ra と rb を順に呼ぶ |
| `rra` / `rrb` | top を `top->prev` に戻す |
| `rrr` | rra と rrb を順に呼ぶ |

#### 操作の出力フォーマット

各操作関数は、実行と同時に**操作名を改行付きで stdout に出力**する。

```
ra\n
pb\n
sa\n
...
```

- 1行に1操作、`\n` 区切り
- 余計な文字（スペース、装飾、ログなど）は一切混ぜない
- checker はこの出力を読み取って判定するため、フォーマット厳守

#### bench対応のための拡張

各操作関数は、実行時に**カウンタもインクリメントする**。詳細は 3-7 を参照。

### 3-4. メイン制御（フラグ解釈と戦略選択）

#### やること

`main` 関数で引数を解釈し、戦略を選んでソート関数を呼び出す。

#### フラグの種類

| フラグ | 動作 |
|-------|------|
| `--simple` | Simple戦略を強制 |
| `--medium` | Medium戦略を強制 |
| `--complex` | Complex戦略を強制 |
| `--adaptive` | Adaptive戦略（デフォルト） |
| `--bench` | 戦略フラグと併用可。ソート後にbench出力 |

#### 戦略フラグの排他性

`--simple` / `--medium` / `--complex` / `--adaptive` は**互いに排他**。複数指定されたらエラーとする。

`--bench` は上記4つのどれとでも併用可能。

#### 処理の流れ

1. 引数からフラグを分離し、残りを整数値として扱う
2. 整数値をパース・検証
3. フラグなし or `--adaptive` → Adaptive戦略を呼ぶ
4. その他のフラグ → 対応する戦略を呼ぶ
5. ソート完了後、`--bench` が指定されていたらbench出力

#### 戦略の切り替え方針

各ソート関数を共通のシグネチャ（例: `void sort_xxx(t_stack *a, t_stack *b)`）で揃え、**関数ポインタ**で切り替える。

理由:
- フラグ解釈と戦略呼び出しを分離できる
- 新しい戦略を追加するときに main の修正が最小限で済む
- テスト時に戦略単体を呼び出しやすい

### 3-5. disorderメトリック

#### やること

スタックaの「乱れ度」を0〜1の数値で表す指標を実装する。Adaptive戦略がどのソートを呼ぶか判断するために使う。また、`--bench` 出力にも使う。
重複しない組み合わせで比較を行っていく。

```
disorder = (逆転対の数) / (全ペア数)
```

- **0** = 完全にソート済み（昇順）
- **1** = 完全な逆順（最悪の乱れ）

#### 逆転対(inversion pair)とは

添字 `i < j` なのに、値 `a[i] > a[j]` となるペアのこと。
つまり「前にあるのに、後ろの要素より大きい」ペア。

例: スタック `[5, 3, 8, 1]`

| ペア | 値 | 逆転? |
|------|-----|------|
| (0, 1) | (5, 3) | ✓ |
| (0, 2) | (5, 8) | × |
| (0, 3) | (5, 1) | ✓ |
| (1, 2) | (3, 8) | × |
| (1, 3) | (3, 1) | ✓ |
| (2, 3) | (8, 1) | ✓ |

逆転対 4個 / 全ペア 6個 = disorder 0.67

#### 全ペア数の計算

n個から2個選ぶ組み合わせ数（ₙC₂）:

```
全ペア数 = n × (n - 1) / 2
```

「2で割る」のは、(i, j) と (j, i) が同じペアなので片方しか数えないため。

#### 採用方針: リスト上で直接計算

500要素レベルではどちらでも体感差はない（数ミリ秒）。disorder計算はソート前に1回だけ走るため、ソート本体の操作数（数千〜数万）に比べれば誤差。シンプルさを優先する。

実装時のループ条件のポイント:
- 外側ループは「i が末尾でない間」（最後の要素を i にしても比較相手がない）
- 内側ループは「j が i の次から、リストを一周するまで」
- 循環リストなので、j は `a->top` に戻ってきたら止める（配列の `j < n` に相当）
- j を `i->next` から開始することで、重複比較と自己比較を防ぐ

#### 計算コスト

- O(n²)。500要素で約125,000回の比較
- ソート前に1回だけ計算するため、全体への影響は無視できる

#### 重要な原則

**disorderはソート開始前に一度だけ計算する**。ソート途中で再計算してはいけない（O(n²)が毎回走るとトータルでO(n³)以上になり、致命的に遅くなる）。

計算結果は、戦略選択と bench 出力の両方で使うため、保存しておく。

### 3-6. ソートアルゴリズム

#### 採用方針サマリ

| 階級 | 採用アルゴリズム |
|------|-----------------|
| Simple (O(n²)) | 挿入ソート（bへ退避型） |
| Medium (O(n√n)) | チャンクソート（√n分割） |
| Complex (O(n log n)) | 基数ソート（LSD, Least Significant Digit） |
| Adaptive | disorderで上3つを切り替え |

各アルゴリズムの採用根拠と不採用候補との比較は **5章** を参照。

#### Adaptiveのしきい値

| disorderの範囲 | 採用戦略 |
|---------------|---------|
| < 0.2 | Simple |
| 0.2 〜 0.5 | Medium |
| ≥ 0.5 | Complex |

####  戦略名と複雑度の記録

Adaptive戦略は内部でSimple/Medium/Complexのいずれかを呼ぶ。`--bench` モードで「どの戦略が実際に使われたか」を出力する必要があるため、**選択した戦略の名前と複雑度クラスをスタック内（または共通の構造体）に保存しておく**。

例: Adaptive がdisorder=0.6でComplexを選んだ場合、benchには `Strategy: complex (O(n log n))` と出力する。

#### 各アルゴリズムの動作概念

##### Simple: 挿入ソート（bへ退避型）

bにソート済み列を作りながら、aから1個ずつ取り出してbの正しい位置に挿入していく方式。

##### Medium: チャンクソート（√n分割）

値の範囲を √n個 のチャンクに分割し、aから各値を「目標チャンクに属するなら pb、違うなら ra」で振り分ける。bがほぼ整列するため、戻すコストが下がる。

##### Complex: 基数ソート（LSD）

1. 全要素にインデックスを振る（最小=0、最大=n-1）
2. インデックスを2進数で表現する（500要素なら9ビット = 2⁹=512で表現可能）
3. 下位ビットから順に、各ビットを見る:
   - ビットが **0** なら pb（bに退避）
   - ビットが **1** なら ra（aに残してずらす）
4. b全体を pa でaに戻す
5. これをビット数（500要素なら9回）繰り返すと完全にソートされている

500要素で4000〜5500操作。優秀ランクに届く。

下位ビットから順に**安定ソート**を繰り返すと、上位ビットでのソート結果が下位の並びを保ったまま整列する、という基数ソートの性質を利用している。

##### Adaptive: 戦略選択

1. disorderを計算
2. しきい値に基づいてSimple/Medium/Complexのいずれかを呼ぶ
3. 呼び出した戦略の名前と複雑度クラスを保存（bench用）

### 3-7. ベンチマークモード(--bench)

#### やること

`--bench` フラグが指定されたとき、ソート完了後にstderrへ実行統計を出力する。stdout（操作列）とstderr（統計）を分離することで、checkerとの組み合わせでも動作する設計とする。

#### 出力する情報

| 項目 | 形式（提案） |
|------|------|
| disorder値 | 小数2桁の％表示（例: `disorder: 47.83%`） |
| 戦略名と複雑度クラス | 例: `Strategy: complex (O(n log n))` |
| 総操作数 | 例: `Total operations: 4823` |
| 各操作の内訳 | sa, sb, ss, pa, pb, ra, rb, rr, rra, rrb, rrr それぞれの回数 |

**注**: 詳細フォーマット（区切り文字、表示順、ラベル名など）は課題文に明示がないため、ペアで合意して統一する。

#### 実装方針

**操作カウンタはスタック構造体に組み込む**。
各操作関数（ra, pa, sb など）は、操作を実行する際に対応するカウンタをインクリメントする。これにより、ソートロジック側はカウンタの存在を意識せずに操作を呼べる。

カウンタの設計:
- 11個の操作それぞれにカウンタを1つずつ持つ
- 総操作数のカウンタも別途持つ（毎回11個合計するのは無駄なため）
- スタックaとbのどちらでも参照できる位置に置く（共有カウンタ）

#### フラグの状態管理

- `main` 関数で `--bench` フラグの有無を判定し、設定用の構造体に保存する
- ソート完了後、フラグが有効ならstderrに出力する関数を呼ぶ
- フラグが無効ならカウンタは更新するが出力はしない（実装簡略化のため）

#### bench出力に必要な情報のまとめ

ソート完了時点で以下が揃っている必要がある:
- disorderの値（3-5で計算済み、保存しておく）
- 使用した戦略の名前と複雑度クラス（3-6で保存）
- 各操作のカウンタ（3-3で更新）
- 総操作数（同上）

#### 使用例

```
./push_swap --bench 4 67 3 87 23 > /dev/null
  → 操作列はstdoutに、bench結果はstderrに出るので、stdoutを捨てればbench結果だけ見える

./push_swap --bench 4 67 3 87 23 2>bench.txt | ./checker 4 67 3 87 23
  → stdoutはcheckerへ流しつつ、stderrのbench結果はファイルに保存
```

#### 実装上の注意

- bench出力は**必ずstderr**へ。stdoutに出すとcheckerが操作列として読んでしまい誤動作する
- `--bench` は他の戦略フラグ（`--simple` など）と併用可能

## 4. データ構造の選定（詳細根拠）

### 選択肢の比較

push_swapの11個の操作（sa/sb/ss/pa/pb/ra/rb/rr/rra/rrb/rrr）を全て効率的に実現できるデータ構造を選ぶ。

| 観点 | 通常の配列 | リングバッファ | 双方向循環連結リスト |
|------|-----------|---------------|---------------------|
| sa/sb (先頭2つswap) | O(1) | O(1) | O(1) |
| pa/pb (要素移動) | O(n) | O(1) | O(1) |
| ra/rb (回転) | **O(n)** | O(1) | O(1) |
| rra/rrb (逆回転) | **O(n)** | O(1) | O(1) |
| メモリ効率 | ◎ | ◎ | △ (ノードあたり16バイト追加) |
| キャッシュ局所性 | ◎ | ◎ | ✗ |
| 実装の素直さ | - | △ (モジュロ計算が必要) | ◎ |
| デバッグしやすさ | 易 | やや難 | 易 |

### 採用方針: 双方向循環連結リスト

理由:

1. **全操作がO(1)で実現できる**。回転系がO(n)になる通常の配列は除外。
2. **コードと仕様が1対1で対応する**。raは「ポインタを1つ進める」、pa/pbは「ノードを切り離して別のリストに繋ぐ」など、操作の意図がそのままコードに現れる。
3. **リングバッファより実装が直感的**。リングバッファも全操作O(1)で実現できるが、pa/pbを実行するたびに「インデックス計算」「マイナス値の補正」「サイズ更新」など手数が多い。双方向リストならポインタの繋ぎ替えだけで完結し、push_swapの操作概念（ノードを取って別のリストに繋ぐ）とコードが直接対応する。
4. **評価で説明しやすい**。ポインタの動きを図で説明でき、リングバッファのモジュロ計算を毎回説明する負担がない。
5. **メモリオーバーヘッドは無視できる規模**。500要素でも追加メモリは約8KB。push_swapの規模では問題にならない。

リングバッファも機能的には等価で、メモリ効率では勝るが、ラップアラウンド計算がコード全体に分散して可読性を下げるため不採用。


## 5. 各ソートアルゴリズムの選定（詳細根拠）

3つの複雑度クラス（O(n²) / O(n√n) / O(n log n)）それぞれに1つずつアルゴリズムを実装する。push_swap言語（11個の操作）との相性と、合格基準を満たす操作数で動くことが必須。

### Simple枠 (O(n²))

| アルゴリズム | 100要素の操作数(目安) | 判定 |
|-------------|---------------------|------|
| バブルソート | 5000超 | 不採用（合格基準オーバー） |
| 選択ソート | 約2500 | 不採用（合格ギリギリで余裕がない） |
| 挿入ソート (bへ退避型) | 約1500 | **採用** |

bにソート済み列を作りながら、aから1個ずつ取り出してbの正しい位置に挿入していく方式。Simple枠の3候補の中で操作数が最も少なく、O(n²)階級としては最適。

### Medium枠 (O(n√n))

| アルゴリズム | push_swapとの相性 | 判定 |
|-------------|------------------|------|
| シェルソート | 離れた位置の比較が必要 (先頭しか触れないpush_swap言語と非整合) | 不採用 |
| チャンクソート | 「振り分け→戻し」の構造とpush_swap言語が一致 | **採用** |

#### なぜチャンク数を √n にするのか

チャンクソートのコスト構造:
- 振り分けコスト ≈ k × n（チャンク数kに比例）
- 戻すコスト ≈ n² / k（チャンクサイズn/kに比例）

総コスト f(k) = kn + n²/k を最小化すると k = √n となる。
直感的には「**チャンク数とチャンクサイズの両方をバランスさせる点が最適**」ということ。

500要素なら √500 ≈ 22個のチャンクに分けるのが理論的に最適。

### Complex枠 (O(n log n))

| アルゴリズム | push_swapとの相性 | 判定 |
|-------------|------------------|------|
| マージソート | 分割した片方の保管場所がない（2スタック制約） | 不採用 |
| ヒープソート | 親子関係のある木構造を2スタックで表現不可能 | 不採用 |
| クイックソート | 分割の再帰で操作数が膨らむ | 不採用 |
| 基数ソート (LSD) | ビットが0ならpb、1ならraだけで動く | **採用** |

push_swap言語と最も相性が良い。詳細な動作は 3-6 を参照。

### しきい値(0.2 / 0.5)の根拠

課題仕様で指定された値。直感的な根拠は以下の通り:

| 範囲 | 戦略 | 根拠 |
|------|------|------|
| < 0.2 | Simple | ほぼ整列済みなら、挿入ソートが余計な操作なしで完了する |
| 0.2 〜 0.5 | Medium | 部分的に整列、チャンクソートが構造を活かして整える |
| ≥ 0.5 | Complex | 完全にランダムに近い、基数ソートで全体再構築が最も効率的 |

### 補足知識

逆転対の数は「**バブルソートで必要な隣接swap回数**」と数学的に一致する。push_swapではより複雑な操作だが、disorderが高いほど必要な操作数が多くなる傾向がある。
