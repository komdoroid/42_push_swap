# Push_swap

*Swap_push だとしっくりこないから。*

**概要:** このプロジェクトでは、限られた命令セットを使用し、可能な限り少ないアクション数で、スタック上のデータをソート（並び替え）させます。

成功するためには、さまざまな種類のアルゴリズムを操作し、最適化されたデータソートに最も適した解決策（多くの選択肢の中から）を選択する必要があります。

* **参加人数:** まさに2名の学習者で完了させるグループアクティビティです。
* **バージョン:** 1.1

## 第I章 はじめに (Foreword)

むかしむかし、コンピュータサイエンスという神秘的な国で、あるドナルド・クヌース（Donald Knuth）¹ が、アルゴリズムがどのようにスケールするかを議論しやすくするために「ビッグO記法（Big-O notation）」を普及させました。

ビッグOとは、簡単に言えば次のようなことを丁寧に伝えるための方法です：

> 「君のコードは、永遠に高速に動き続けるか……さもなければ、待っている間にコーヒーを淹れて飲み、老け込んでしまうほど遅くなるか、のどちらかだ。」

伝説によると、初期のプログラマーにはビッグOがありませんでした。彼らはただコードを実行し、もし時間がかかりすぎたらハードウェアのせいにしていました。その後、ビッグOが登場し、「入力が増えると、CPUの回し車にどれだけハムスターを投入しようが、君のアルゴリズムが根本的にクソである」ということを数学的に証明してしまい、楽しみを台無しにしました。

なぜこれが `push_swap` に関係するのでしょうか？なぜなら、今あなたは2つのスタックと、不器用で制限された一連の移動コマンドだけを武器に、睡眠不足の挿入ソートよりも速く数値をソートしようとしているからです。

ビッグOはあなたに残酷な真実を突きつけます。入力が大きくなると、優れた $\mathcal{O}(n \log n)$ の戦略は、あなたの不器用な $\mathcal{O}(n^2)$ の戦略よりも常に長生きします。……もちろん、あなたが数学を無視して、操作カウントが急上昇するのを眺めるのが好きなら話は別ですが。

アルゴリズムを設計する際は、覚えておいてください。ビッグOはあなたを脅すためにあるのではなく、次のようなコードを：

```text
pb; pa; pb; pa; pb; pa; ...

```

連続で10,000回も書き連ねて、「なぜチェッカーに嫌われるんだろう」と悩む人間になるのを防ぐためにあるのです。賢くソートしましょう。遅くではなく。

*ビッグOはポップコーンのサイズ評価ではありません……が、もしそうだとしても、O(n log n)が一番美味しいところ（スイートスポット）です。*

¹ *あの見事な髭を持ち、無限の忍耐力を持つお方のことです。*

`Push_swap` プロジェクトは、非常にシンプルで高度にストレートなアルゴリズムプロジェクトです。データをソートしなければなりません。

## 第V章 目的 (Objectives)

このプロジェクトの目的は、非常に具体的な方法でアルゴリズムの複雑さ（計算量）を発見してもらうことです。

数値をソートするのは簡単ですが、たった2つのスタックと一握りの許可された移動だけで高速にソートするのは全く別の話です。また、完全にランダムなリストをソートすることと、ほとんどソートされているリストをソートすることも、極めて異なります。

アルゴリズムの選択が、素早い勝利と、終わりのない操作スクロールのどちらをもたらすかの分かれ目になることを、すぐに実感することでしょう。

### VI.3 アルゴリズムの要件

アルゴリズムの複雑さ（時間と空間）を厳密に理解させるため、**4つの異なるソート戦略**を実装し、それらを `push_swap` プログラムに統合しなければなりません。プログラムは、実行時（ランタイム）に入力構成に基づいて戦略を選択できなければなりません。

#### VI.3.1 計算量モデルと操作制約

すべての戦略はC言語で実装され、ソートを実行するための Push_swap 命令のシーケンスを生成しなければなりません。
これは以下を意味します：

* あなたのCアルゴリズムは入力を分析し、適切な命令のシーケンス（`sa`, `sb`, `ss`, `pa`, `pb`, `ra`, `rb`, `rr`, `rra`, `rrb`, `rrr`）を生成します。
* 戦略の出力は、スタックをソートするこれらの命令のシーケンスです。
* **計算量クラスを主張する場合、それは古典的な配列ベースのアルゴリズムの理論的複雑さではなく、生成された「Push_swap命令の数」で測定されたコストを反映していなければなりません。**

#### VI.3.2 無秩序度メトリック (Disorder metric) 【必須】

この課題において、無秩序度（disorder）とは、初期スタック `a` がソートされた状態からどれだけ離れているかを示す **0 から 1 までの数値**です。
数字がすでに正しい順序にある場合、無秩序度は `0` です。最悪の順序にある場合、無秩序度は `1` です。その間の値は、スタックが部分的にソートされているものの、まだ乱れていることを意味します。

これを計算するには、スタック内の数字の可能なペアをすべて見ることを考えます。
大きな数字が小さな数字の前に現れるたびに、そのペアは「間違い（誤り）」としてカウントされます。間違いが多いほど、無秩序度は `1` に近づきます。

```text
function compute_disorder(stack a):
    mistakes = 0
    total_pairs = 0
    for i from 0 to size(a)-1:
        for j from i+1 to size(a)-1:
            total_pairs += 1
            if a[i] > a[j]:
                mistakes += 1
    return mistakes / total_pairs

```

**いかなる移動（ムーブ）を行う前にも、無秩序度を測定しなければなりません。**

#### VI.3.3 要求される4つの戦略

1. **シンプルアルゴリズム ($\mathcal{O}(n^2)$):**
$\mathcal{O}(n^2)$ クラスのベースラインアルゴリズムを少なくとも1つ実装してください。
*例: 挿入ソートの適応、選択ソートの適応、バブルソートの適応、単純な最小/最大抽出法など。*
2. **ミディアムアルゴリズム ($\mathcal{O}(n\sqrt{n})$):**
$\mathcal{O}(n\sqrt{n})$ クラスのアルゴリズムを少なくとも1つ実装してください。
*例: チャンクベースのソート（$\sqrt{n}$ 個のチャンクに分割）、ブロックベースのパーティショニング法、$\sqrt{n}$ 個のバケットを使用したバケットソートの適応、範囲ベースのソート戦略など。*
3. **複雑なアルゴリズム ($\mathcal{O}(n \log n)$):**
$\mathcal{O}(n \log n)$ クラスのアルゴリズムを少なくとも1つ実装してください。
*例: 基数ソート（Radix sort）の適応（LSDまたはMSD）、2つのスタックを使用したマージソートの適応、スタックパーティショニングを伴うクイックソートの適応、ヒープソートの適応、フェニック木（Binary Indexed Tree）アプローチなど。*
4. **カスタム適応型アルゴリズム (学習者独自の設計):**
測定された無秩序度に応じて異なる内部メソッドを選択する、適応型（アダプティブ）戦略を設計してください。特定の名前の付いたアルゴリズムに縛られる必要はなく、内部のテクニックは完全に自由です。
ただし、設計は **Push_swap命令モデルにおける** レジームごとの以下の計算量ターゲットを尊重しなければなりません：
* **低無秩序度:** 無秩序度 $< 0.2$ の場合、選択したメソッドは $\mathcal{O}(n^2)$ 時間で動作しなければならない。
* **中無秩序度:** $0.2 \le$ 無秩序度 $< 0.5$ の場合、選択したメソッドは $\mathcal{O}(n\sqrt{n})$ 時間で動作しなければならない。
* **高無秩序度:** 無秩序度 $\ge 0.5$ の場合、選択したメソッドは $\mathcal{O}(n \log n)$ 時間で動作しなければならない。



リポジトリ（例: `README.md`）に、しきい値の根拠、各レジームで使用される内部テクニック、およびPush_swapモデル内での時間と空間の簡潔な計算量引数（上限）をドキュメント化しなければなりません。

### VI.5 `push_swap` プログラム

| 項目 | 内容 |
| --- | --- |
| **プログラム名** | `push_swap` |
| **提出ファイル** | `Makefile`, `*.h`, `*.c` |
| **Makefileルール** | `NAME`, `all`, `clean`, `fclean`, `re` |
| **引数** | スタック `a`: 整数のリスト |
| **許可される外部関数** | `read`, `write`, `malloc`, `free`, `exit`, `ft_printf` またはあなたがコードした同等のもの |
| **Libftの使用** | 許可されている |
| **説明** | スタックをソートする |

プロジェクトは以下のルールに準拠しなければなりません：

* ソースファイルをコンパイルする Makefile を提出しなければなりません。再リンクしてはなりません。
* **グローバル変数の使用は禁止されています。**
* 引数として以下を受け取る `push_swap` という名前のプログラムを書かなければなりません：
* 整数のリストとしてフォーマットされたスタック `a`（最初の引数がスタックのトップになります）。
* **オプションの戦略セレクター:**
* `--simple` : $\mathcal{O}(n^2)$ アルゴリズムの使用を強制する。
* `--medium` : $\mathcal{O}(n\sqrt{n})$ アルゴリズムの使用を強制する。
* `--complex`: $\mathcal{O}(n \log n)$ アルゴリズムの使用を強制する。
* `--adaptive`: 無秩序度に基づいた適応型アルゴリズムの使用を強制する。**（セレクターが指定されない場合のデフォルトの挙動）**




* プログラムは、スタック `a` をソートするための**可能な限り最小の Push_swap 操作リスト**を表示しなければなりません。最も小さい数がスタックのトップにくる必要があります。
* 操作は `\n` のみで区切られなければなりません。
* 各アルゴリズムに対して主張される計算量クラスは、このモデルで有効でなければなりません。
* 戦略の選択は、すべての有効な入力に対して機能しなければなりません。セレクターフラグは、入力のサイズや無秩序度に関係なく動作する必要があります。
* パラメータが指定されない場合、プログラムは何も表示せず、プロンプトをそのまま返さなければなりません。
* エラーの場合、標準エラーに `Error` に続けて `\n` を表示しなければなりません。エラーには、たとえば、整数ではない引数、有効な範囲を超えた整数、重複した値などがあります。
* バイナリには 4 つの戦略（Simple, Medium, Complex, Adaptive）すべてを組み込む必要があります。選択された戦略名と計算量クラスは、`--bench` モードで利用可能でなければなりません。
* **オプションのベンチマークモード (`--bench`)** は、ソート後に以下を表示しなければなりません：
* 計算された無秩序度（小数点以下2桁の % 表示）。
* 使用された戦略の名前とその理論的計算量クラス。
* 総操作数。
* 各操作タイプ（`sa`, `sb`, `ss`, `pa`, `pb`, `ra`, `rb`, `rr`, `rra`, `rrb`, `rrr`）のカウント。
* **ベンチマーク出力は stderr（標準エラー出力）に送信され、フラグが存在する場合にのみ表示されなければなりません。**

### VIII.1 The "checker" program

| 項目 | 内容 |
| --- | --- |
| **プログラム名** | `checker` |
| **提出ファイル** | `*.h`, `*.c` |
| **Makefileルール** | `bonus` |
| **引数** | スタック `a`: 整数のリスト |
| **許可される外部関数** | `read`, `write`, `malloc`, `free`, `exit`, `ft_printf` またはあなたがコードした同等のもの |
| **Libftの使用** | 許可されている |
| **説明** | ソート操作を実行する |

* 引数として整数のリストとしてフォーマットされたスタック `a` を受け取る `checker` という名前のプログラムを書きます。最初の引数がスタックのトップにくる必要があります（順序に注意してください）。引数が与えられない場合は、停止して何も表示しません。

* エラーの場合、標準エラーに `Error` に続けて `\n` を表示しなければなりません。エラーには、たとえば、一部の引数が整数ではない、一部の引数が整数の限界を超えている、重複がある、命令が存在しない、および/またはフォーマットが正しくない、などが含まれます。

```bash
$> ./checker 3 2 1 0
rra
pb
sa
rra
pa
OK
$> ./checker 3 2 1 0
sa
rra
pb
KO
$> ./checker 3 2 one 0
Error
$> ./checker "" 1
Error
$>

```

*提供されたバイナリとまったく同じ挙動を再現する必要はありません。エラーを管理することは必須ですが、引数をどのようにパースするかはあなた次第です。*
